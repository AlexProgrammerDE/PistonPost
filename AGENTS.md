# Repository Guidelines

## Start here

Read PLAN.md before making changes. It is the source of truth for scope, architecture, phase order, acceptance criteria, and remaining work.

When implementing the plan:

- Work from the first incomplete phase unless the user gives a narrower task.
- Mark completed checklist items in PLAN.md only after their exit criteria pass.
- Keep the repository usable at the end of every phase.
- Verify current upstream documentation before relying on beta or Cloudflare-specific APIs.
- Do not silently expand product scope. Record meaningful deviations in PLAN.md.
- Preserve unrelated user changes in a dirty worktree.

## Repository structure

The initialized repository currently contains:

- apps/web/: the TanStack Start application and the only deployable Cloudflare Worker.
- packages/ui/: shared shadcn/ui components, Base UI primitives, fonts, styles, and UI utilities.
- PLAN.md: the executable rewrite and migration plan.
- AGENTS.md: repository-wide contributor and agent rules.
- CLAUDE.md: Claude entrypoint that imports this file.

The target structure adds these workspaces as their phases begin:

- apps/migrate/: one-time, resumable legacy MongoDB and media migration CLI.
- packages/auth/: Better Auth server factory, client, authorization helpers, and email contracts.
- packages/db/: D1 Drizzle schema, generated migrations, database factory, and repositories.
- packages/domain/: product schemas, policies, service contracts, and shared domain types.
- packages/email/: React Email templates and Cloudflare email transport.
- packages/config/: shared TypeScript and Oxc configuration where centralization is useful.

Do not create a package until it owns a real boundary. Keep Cloudflare runtime bindings and request handling in apps/web. Keep packages/ui free of PistonPost domain behavior.

## Development commands

Install dependencies once from the repository root:

```bash
bun install
```

Current scaffold commands:

- bun run dev: start the TanStack Start development server through Turborepo.
- bun run build: build all buildable workspaces.
- bun run lint: run the current scaffold lint task.
- bun run typecheck: type-check all workspaces.
- bun run format: format the current scaffold.

Phase 1 replaces ESLint and Prettier with Oxc and adds these root commands:

- bun run check: run Oxlint, Oxfmt verification, typecheck, and focused static checks.
- bun run fix: apply Oxc lint and formatting fixes.
- bun run test: run workspace tests through Turborepo.
- bun run ci: run the complete local CI gate.
- bun run dev:web, build:web, test:web, and typecheck:web: package-specific shortcuts.

Prefer root tasks for normal work. Run package-local commands when local configuration, test preload, Wrangler binding generation, or Drizzle configuration matters.

## Package management and generated files

- Use Bun only. Do not add npm, pnpm, or Yarn lockfiles.
- Use bunx --bun for one-off package CLIs.
- Keep bun.lock committed.
- Use Turborepo for root task orchestration and cache only deterministic tasks.
- Generated route trees, Worker types, Better Auth schemas, and Drizzle migrations must be generated by their owning tools.
- Never hand-edit generated files to make a check pass.

## TypeScript and React

- Use strict TypeScript. Avoid any, unsafe casts, and non-null assertions unless the boundary is proven and documented.
- React components use PascalCase filenames and exports.
- Hooks and utilities use kebab-case filenames.
- Prefer named exports for reusable code.
- Keep server-only modules visibly separated and never import them into client bundles.
- Parse untrusted data at the boundary with Zod or the owning library schema.
- Model domain states with discriminated unions instead of loosely related booleans.
- Prefer composition over boolean-prop-heavy components.
- Use React 19 APIs when they solve the actual state or scheduling problem.
- Use useActionState and useFormStatus for server-action-style forms when TanStack Form is not the better fit.
- Use useOptimistic for reaction, comment, and reorder experiences that have a clear rollback path.
- Use useTransition or startTransition for non-urgent navigation and filter updates.
- Use useDeferredValue for expensive feed or table filtering.
- Use useEffectEvent for event bridges that need fresh state without resubscribing.
- Use useSyncExternalStore for external stores.
- Use Activity only when hidden UI must preserve state and reduce work.
- Do not add memo, useMemo, or useCallback without a measured or structurally clear reason.

## TanStack conventions

- TanStack Router owns route definitions, validation, loaders, pending states, and route-level errors.
- TanStack Query owns server-state caching, invalidation, optimistic updates, and mutation status. Do not mirror query data into local state.
- TanStack Form owns complex product forms and validation. Keep schemas reusable between client and server.
- TanStack Table v9 is beta. Pin its exact version and update it deliberately.
- Build shared table features once in a focused table module. Do not duplicate sorting, filtering, selection, pagination, or column-visibility state across screens.
- Keep table state in the v9 atom/store model expected by the installed version.
- Use stable row IDs from domain records.
- Test row expansion, selection, pagination, URL state, and mutation invalidation at the shared table boundary.

## UI and design

The design direction is editorial and media-first with a restrained industrial PistonPost character. Use the generated Maia preset, taupe theme, amber charts, Outfit body type, Raleway headings, and Hugeicons. Avoid generic dashboard card grids, decorative gradients, glass effects, oversized marketing headings, and excessive rounded containers.

- Use shadcn/ui components idiomatically before writing custom primitives.
- This project uses Base UI, not Radix. Use render props where Base UI expects them and do not use asChild.
- Add components through the shadcn CLI from apps/web so they land in packages/ui.
- Inspect shadcn component docs and examples before composing unfamiliar components.
- Fix consumer composition first. Modify packages/ui/src/components only when the shared primitive itself is demonstrably wrong.
- Use semantic Tailwind tokens such as bg-background and text-muted-foreground.
- Use gap-* utilities. Never use space-* utilities.
- Use the shared cn utility for conditional class names.
- Do not manually set icon width or height inside controls that already style icons.
- Use Hugeicons from the configured packages. Do not introduce another general icon library.
- Use Field, FieldGroup, FieldLabel, FieldDescription, and FieldError for forms.
- Use full Card composition when a card is semantically appropriate.
- Use Sonner for transient notifications, Skeleton for loading placeholders, Empty for empty states, Alert for actionable errors, and Dialog or AlertDialog for confirmations.
- Dialog, Sheet, and Drawer content must have accessible titles and descriptions.
- Every input needs an associated label.
- Icon-only controls need accessible names and tooltips where the meaning is not obvious.
- Keep focus indicators visible and respect reduced motion.
- Do not use array indexes as React keys.
- Use generateN(count) for identity-safe skeleton sequences:

```ts
export function generateN(count: number) {
  return Array.from({ length: Math.max(0, Math.floor(count)) }, (_, index) => index + 1)
}
```

## Better Auth

- Use Better Auth with the Drizzle adapter and D1.
- Reuse the Better Auth UI architecture proven in EnderDash through @better-auth-ui/core and @better-auth-ui/react, adapted to this Base UI project.
- Do not copy EnderDash organizations, subscriptions, API keys, OAuth provider, or Minecraft-specific behavior.
- Do not use Better Auth active organizations. PistonPost has no organization domain.
- Create the auth instance from request-scoped Cloudflare bindings. Do not capture request bindings in module-global mutable state.
- Keep authorization separate from authentication. Domain services must still check ownership, role, visibility, and moderation policy.
- Use short-lived encrypted cookie cache only after tests prove revocation and role changes behave correctly.
- Keep auth, session, and API responses private and no-store.
- Protect sign-in, sign-up, password reset, magic-link, OTP, and sensitive mutation endpoints with Turnstile and rate limits.
- Never log tokens, secrets, email message bodies, session cookies, or full IP addresses.

## Cloudflare runtime

- apps/web is the only production Worker unless PLAN.md is explicitly changed.
- Keep wrangler.jsonc valid and generate binding types after every binding change.
- Use a current compatibility date and nodejs_compat only where dependencies require it.
- Copy EnderDash configuration patterns selectively. Do not copy account IDs, resource IDs, secrets, hostnames, or irrelevant bindings.
- Use D1 for relational data, R2 for original files, Cloudflare Images for image transformation and delivery, and Cloudflare Stream for video.
- Use Queues for durable asynchronous work and Workflows for multi-step, resumable operations.
- Use Workers Cache only for explicitly public responses. Authenticated, personalized, mutation, preview, draft, and unlisted responses must not be publicly cached.
- Prefer Secrets Store bindings for production secrets. Use local development variables only in ignored files.
- Validate upload size, MIME type, extension, dimensions, ownership, and state on the server.
- Never expose write-capable storage credentials to the browser.
- Apply rate limits at both broad request and sensitive action boundaries.
- Keep analytics free of message content, email addresses, tokens, and other direct personal data.

## Database and Drizzle

- D1 and SQLite constraints are the production truth. Do not design against PostgreSQL-only behavior.
- Define schema in packages/db/src/schema first.
- Never manually edit files in drizzle/ or drizzle/meta/.
- Generate migrations from the owning workspace:

```bash
bunx drizzle-kit generate --name <migration-name>
```

- Commit generated migration artifacts exactly as Drizzle produces them.
- Add foreign keys, unique constraints, check constraints, and indexes intentionally.
- Use transactions for domain changes that must be atomic.
- Use cursor pagination for feeds and large tables.
- Avoid unbounded joins and N+1 queries.
- Keep migration scripts idempotent and record source IDs in explicit legacy mapping columns or tables.

## Testing

- New behavior should include targeted unit or integration tests when practical.
- Test domain policy separately from UI.
- Test D1 repositories against a real SQLite-compatible test database, not mocked query builders.
- Test Better Auth request scoping, session behavior, email dispatch, captcha failures, and authorization boundaries.
- Test upload initiation, completion, validation, orphan cleanup, and unauthorized access.
- Test public, unlisted, draft, deleted, and moderated visibility separately.
- Test migration reruns, partial failures, count reconciliation, ID preservation, and media checksums.
- Use Playwright for critical journeys: sign in, create each post type, edit, react, comment, manage settings, moderate, and sign out.
- A phase is not complete until its listed exit criteria pass.

## Git and commits

- Never create a branch unless the user explicitly asks.
- Use Conventional Commit messages in the form type(scope): subject.
- Prefer feat, fix, docs, refactor, test, chore, ci, build, and perf.
- Keep the subject concise and imperative.
- Keep commits focused and stage only intended files.
- Never bypass hooks.
- Never use -n or any other option to skip Lefthook or Git hooks.
- Wait for hooks to finish and fix failures at their source.
- Do not rewrite or discard user changes.

## Documentation

- Keep PLAN.md current as phases land.
- Update README.md when setup or top-level commands change.
- Document binding names and required secret names, but never secret values or Cloudflare resource IDs.
- Use natural, direct language. Do not use em dashes.
- Prefer short paragraphs, descriptive links, fenced commands, and compact tables.
- Record architectural decisions that future agents might otherwise reverse accidentally.
